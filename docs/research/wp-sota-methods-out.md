**Critical details by family**

- **Docker Sandboxes credential proxy**: strong example of the exact design. Secrets are stored host-side, sandbox sees placeholders, proxy substitutes for configured hosts; SSH agent forwarding keeps private keys host-side but lets sandbox request signatures. Registry creds are an exception because they may be written into the VM. Source: https://docs.docker.com/ai/sandboxes/security/credentials/
- **Cloud metadata services**: IMDS/GCP/Azure solve “no static key on disk,” not “compromised workload can’t get a token.” Azure explicitly notes all code on the VM can request tokens for available managed identities. Sources: AWS IMDSv2 https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/configuring-instance-metadata-service.html, Azure managed identity https://learn.microsoft.com/en-us/entra/identity/managed-identities-azure-resources/how-to-use-vm-token
- **Vault Agent vs Secretless**: Vault Agent Injector usually writes secrets into `/vault/secrets`, so a compromised app can read them. Secretless Broker keeps connection secrets in broker process and pipes traffic. Sources: https://developer.hashicorp.com/vault/docs/deploy/kubernetes/injector, https://docs.cyberark.com/conjur-open-source/latest/en/content/overview/scl_how_it_works.htm
- **Request signing nuance**: a proxy can be redesigned as a signer, but then the workload must send unsigned canonical request intent to that signer and the signer must enforce policy before signing. That is not ordinary “inject API key into header.”

**Ranked Bottom Line**

1. **Best SOTA for a compromised local AI coding agent**: sandbox/namespace or VM isolation + deny-by-default egress + host-side credential broker/proxy injection. This is the strongest composable architecture because the key never enters the agent and the agent has no arbitrary network path.

2. **Best upgrade when vendors support it**: workload identity federation / STS-style token exchange, performed by the host proxy, producing short-lived scoped credentials per task/repo/tool. OpenAI and Anthropic WIF, GitHub App installation tokens, AWS STS, SPIFFE/SPIRE, and OIDC federation are the direction of travel.

3. **Necessary backstops**: OS/hardware keystores for host-side storage, HSM/TPM/YubiKey for non-exportable signing keys, audit logs, approval gates for destructive actions, and MCP/OAuth per-tool scoping. These reduce blast radius but do not replace egress containment.

4. **Methods that cannot keep a usable credential out of the process by themselves**: env vars, `.env`, config files, Vault Agent file templates, Kubernetes Secrets, cloud metadata tokens delivered directly to workload, OAuth access tokens inside the agent, fine-grained PATs inside the agent. They improve operations or blast radius, but a compromised agent can read/use them.

5. **Request-signing and mTLS credentials are special**: generic header injection does not work. Either keep the signing key outside and expose a narrow signing/connection broker with strict policy, or accept that the client process holds signing capability and make sandbox + egress + short lifetime your backstop.

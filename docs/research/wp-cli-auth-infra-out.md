**Why SigV4 is class C:** AWS says SigV4 creates a canonical request, calculates a signature using AWS credentials, and adds that signature to the `Authorization` header; AWS then recomputes and verifies it. The secret is not just a transport header. It is used client-side to derive a per-request signature over method/path/query/headers/body hash/time/region/service. A header-injecting MITM proxy cannot keep the secret out of the CLI while allowing arbitrary signed AWS/S3 calls, unless the proxy itself becomes a full AWS-signing proxy and holds the signing secret. That changes the trust boundary: the CLI no longer talks directly to AWS/S3 auth; the proxy must canonicalize and sign every request correctly, including payload hashes and streaming/chunked signatures.

**SOTA alternative for class C:** use egress containment plus short-lived/scoped credentials: AWS STS temporary credentials, IAM roles, IAM Identity Center/SSO, web identity/OIDC, EC2/ECS metadata roles, or IAM Roles Anywhere for X.509-based exchange into temporary SigV4 credentials.

One-line proxy notes:

- `hcloud`: proxy can inject the Hetzner bearer token; it cannot reduce the server-side power of a broad token, so use project/read-only scoping.
- `s5cmd`/S3: proxy cannot merely inject AWS keys because requests must already be signed; use STS/roles/scoped S3 credentials.
- `wrangler` OAuth: proxy can observe/inject API bearer calls only after OAuth, but the local OAuth session/refresh material is the thing Wrangler manages.
- `wrangler` token: proxy can inject `CLOUDFLARE_API_TOKEN` for Cloudflare API calls; use scoped/TTL/IP-restricted tokens.
- `flarectl`: proxy can inject `CF_API_TOKEN` or legacy key headers; prefer scoped API tokens over global keys.
- `runpodctl`: proxy can inject the Runpod API key for GraphQL calls; use restricted/read-only keys where possible.
- `aws`: proxy cannot replace SigV4 secrets with dummy client creds; use STS, IAM roles, IAM Roles Anywhere, and network egress policy.
- `gh`: OAuth login is local-session oriented; PAT/env mode is injectable bearer-token behavior, but fine-grained/short-lived tokens are the better limiter.
